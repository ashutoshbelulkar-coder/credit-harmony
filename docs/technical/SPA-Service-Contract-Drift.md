# SPA service layer vs Spring Boot — contract drift

**Date:** 2026-03-31  
**Status:** **Spring is the canonical backend** (Vite default proxy **8090**). **Major parity gaps** called out in older revisions (member sub-resources, **POST** API keys, **POST** user deactivate, drift list, overview charts) are **closed** on Java — see [Spring-SPA-Route-Inventory.md](./Spring-SPA-Route-Inventory.md). **List/read JDBC** for consortiums, products, reports, SLA configs, alert rules, users, and audit logs is aligned to **`create_tables.sql`** / **`seed_data.sql`** (see [Technical-Decision-Log.md](./Technical-Decision-Log.md) **TDL-018**). This file tracks **remaining** smaller deltas (e.g. roles **`permissions`** matrix shape, monitoring KPI field names).

**Audience:** Engineers aligning the SPA with **`backend/` (Spring)** or auditing legacy Fastify behaviour.

The SPA [`src/lib/api-client.ts`](../../src/lib/api-client.ts) and [`src/services/`](../../src/services/) were originally authored against **Fastify** (`server/src/index.ts`). The following rows may still apply until fully ported or adapted on the client.

| Topic | Fastify (legacy `server/`) | Spring (`backend/`) | Action if using Spring |
|-------|------------------------|------------------------|-------------------------|
| Port (default) | 8091 (legacy Fastify) | 8090 (canonical Spring) | Default proxy is Spring; override `VITE_API_PROXY_TARGET` for Fastify |
| Institution update | `PATCH /api/v1/institutions/:id` | **`PATCH` supported** on Spring (`InstitutionController`) | SPA uses PATCH |
| Institution list page | `{ content, totalElements, totalPages, page, size }` | Spring `Page` uses **`number`** instead of **`page`** | **`fetchInstitutions`** maps `number` → `page` |
| Institution list `role` query | Fastify filters by **`role=dataSubmitter`** / **`role=subscriber`**. | **`role` query supported** on Spring (`InstitutionRepository`) | Verify edge cases |
| User deactivate | `POST /api/v1/users/:id/deactivate` | **`POST` implemented** (`UserController`); sets `user_account_status` to **`deactivated`** (not soft-delete). **`DELETE /api/v1/users/:id`** remains **Super Admin** soft-delete only | HTTP security allows **GET/PATCH/POST** on `/api/v1/users/**` for roles aligned with `@PreAuthorize`; **DELETE** still global **SUPER_ADMIN** rule |
| User PATCH | Full patch for roles, institution, names | Partial (`UserController` patch subset) | Extend Spring or narrow client |
| JWT roles in payload | e.g. `"Super Admin"`, `"Viewer"` | Often `ROLE_SUPER_ADMIN` style | Align `AuthContext` / guards |
| GET `/api/v1/roles` | Each row includes `permissions` as section×action matrix (Fastify in-memory) | JPA `Role` entity: `id`, `roleName`, `description`, `createdAt` only — **no** matrix JSON | SPA applies `mergeRolePermissionsFromApi`; persist matrix in Spring if the portal must be DB-authoritative for RBAC |
| Monitoring KPIs | `MonitoringKpis` from seed / computed | `MonitoringController` uses DB column names / windows | Normalize DTO or map in `fetchMonitoringKpis` |
| Institution sub-resources | consent, api-access, billing PATCH, product subscriptions, consortium memberships, documents | **Implemented** on Spring (`InstitutionController`) with camelCase-oriented JDBC/JSON responses | Keep `institutions.service.ts` in sync; re-seed SQLite if new columns were added |
| Member Overview charts | `GET /api/v1/institutions/:id/overview-charts` | **Implemented** on Spring (JDBC aggregates, last 30d) | Match Fastify semantics where possible; empty series for new members with no traffic |
| API keys — create | `POST /api/v1/api-keys` | **Implemented** (`ApiKeyController`); body `institutionId`, `environment` | `api_keys` table includes `environment`, `rate_limit_per_minute`; recreate dev DB if schema upgraded |
| Data ingestion drift | `GET /api/v1/data-ingestion/drift-alerts` | **Implemented** (`DataIngestionController`); table `ingestion_drift_alerts` + optional filter via `schema_mapper_registry` payloads | Seed in `seed_data.sql`; run `npm run check:route-parity` for heuristic Fastify vs Spring scan |
| Consortiums API | **`GET/POST/PATCH …/consortiums`** — public consortium JSON is **`id`, `name`, `status`, `membersCount`, `dataVolume`, `description`, `createdAt`**; **`type`**, **`purpose`**, **`governanceModel`** stripped on read and ignored on write; **`GET …/consortiums/:id/members`** returns **`id`, `institutionId`, `institutionName`, `joinedAt`** (no **`role`**); **`members`** on create/patch is **`{ institutionId }[]`**; **`activeConsortiums`** in form-metadata is **`{ id, name }`**; institution **`consortium-memberships`** omits **`consortiumType`** (still includes **`memberRole`** for the institution↔consortium link) | Spring **`ConsortiumController`** uses **JdbcTemplate** against canonical DDL: list/detail alias **`consortium_name` → `name`**, expose **`type`** / **`governanceModel`** / **`description`** from real columns; **`GET …/{id}/members`** does not filter on a non-existent **`is_deleted`** on **`consortium_members`**. **POST** persists **`consortium_code`**, **`consortium_name`**, **`consortium_type`**, **`consortium_status`** (`pending` for new rows), optional **`purpose`** / **`governance_model`** / **`description`**. | SPA may show extra fields vs Fastify-trimmed legacy contract; treat Spring as authoritative for dev SQLite |
| Products list `type` | Fastify / mock often used a category-style **`type`** | Spring list/detail map **`coverage_scope`** (and filters) to JSON **`type`** — there is **no** **`product_type`** column in **`products`**. | Client filters expecting legacy type strings may need to use **`product_code`** or **`coverage_scope`** values from the API |
| Reports list | Fastify in-memory shapes | Spring **`ReportController`** uses **`report_type`**, **`date_range_start` / `date_range_end`** (→ **`dateFrom` / `dateTo`**), no **`is_deleted`** on **`reports`**; cancel maps to **`failed`** (schema CHECK allows only **`queued` \| `processing` \| `completed` \| `failed`**). | Match UI labels to lowercase statuses or normalise in the client |
| SLA / alert rules lists | Legacy column names in older Spring stubs | **`SlaConfigController`**: **`is_active=1`**, order by **`sla_domain`, `metric_name`**. **`AlertRuleController`**: **`ORDER BY rule_name`**; mutate **`alert_rule_status`** **`enabled`/`disabled`**. | — |
| Users list JSON | JPA **`User`** entity (`UserDetails` + lazy roles) | Spring **`GET/PATCH`/invite** return **JdbcTemplate** maps: **`roles[]`** from **`group_concat(role_name)`**, boolean **`mfaEnabled`**, no Hibernate graph on list/read. | **Aligned** with SPA **`UserResponse`** shape |
| Audit logs JSON | Nested `user` vs flat | Spring **`GET /api/v1/audit-logs`** returns flat **`userId`**, **`userEmail`** per row; supports **`entityType`** (and **`entityId`**, **`from`/`to`**, etc.). | **Aligned** with **`auditLogs.service.ts`** **`AuditLogEntry`** |
| `@AuthenticationPrincipal` | Legacy stubs used JPA **`User`** | Controllers and **`AuthController` `/me` + logout** use **`AuthUserPrincipal`**; **`AuditService.log(AuthUserPrincipal, …)`** resolves a JPA reference for **`audit_logs.user_id`**. | — |
| Register member — wizard metadata | `GET …/form-metadata?geography=` as above | **Implemented** — classpath [`config/institution-register-form.json`](../../backend/src/main/resources/config/institution-register-form.json) (keep in sync with SPA `src/data/institution-register-form.json`); **`POST …?geography=`** validation may still lag Fastify | Extend Spring POST validation to match Fastify rules |
| Product packet catalogue | `GET /api/v1/products/packet-catalog` → `{ options }` from **`data-products.json`** (includes **`derivedFields`**) | Same route on **`ProductController`**; body from **`catalog/product-packet-catalog.json`** — **must stay in sync** with **`src/data/data-products.json`** `productCatalogPacketOptions` | Update both JSON sources when the catalogue changes |
| RBAC | Bearer present → allow | `@PreAuthorize` per route | Enforce same rules in Fastify or test Spring-only |
| **Schema Mapper Agent** | `GET/POST/PATCH … /api/v1/schema-mapper/*`, **`GET …/wizard-metadata`** (Step 1 dropdowns), async mapping worker, approvals via `approval_queue` + `entity_ref_id` | **Implemented** on Spring (`SchemaMapperController`, `schema_mapper_*` JSON tables); legacy Fastify in `server/` for comparison | Keep `src/data/schema-mapper.json` in sync with `backend/.../config/schema-mapper.json` |
| Institution single-string label (pickers, dashboard `member`, etc.) | Older Fastify behaviour copied legal into `tradingName` on create; some UIs used `tradingName ?? name` | Spring + current SPA use **legal first** (`COALESCE(NULLIF(TRIM(name),''), trading_name)`; `institutionDisplayLabel`) | **Aligned** on current `server/` + `backend/` — see [API-UI-Parity-Matrix.md](./API-UI-Parity-Matrix.md) § *Institution display labels* |
| **Batch job detail (execution console)** | **`GET /api/v1/batch-jobs/:id/detail`** → `state.batchDetails[id]` or `{ batchId, stages: [] }` (in-memory, string **`batch_id`**) | **JdbcTemplate** + **`batch_phase_logs` / `batch_stage_logs` / `batch_error_samples`**: structured **`phases`**, **`stages`**, **`flowSegments`**, **`logs`**, **`errorSamples`** when phases exist; numeric job **id** in path | SPA **`resolveBatchConsoleData`** + **`useBatchDetail`**; mock **`batchConsoleByBatchId`** for **`BATCH-*`** list ids — see [Canonical-Backend.md](./Canonical-Backend.md) |

**Authoritative Fastify surface:** hand-maintained snapshot [openapi-hcb-fastify-snapshot.yaml](./openapi-hcb-fastify-snapshot.yaml) and live [API-UI-Parity-Matrix.md](./API-UI-Parity-Matrix.md).

**Recommendation:** Treat **Spring** as the **SPA contract** going forward; regenerate or replace [openapi-hcb-fastify-snapshot.yaml](./openapi-hcb-fastify-snapshot.yaml) from Spring OpenAPI when ready. Use Fastify only as a legacy reference.
