# AGENTS.md

## Cursor Cloud specific instructions

### Product overview

HCB (Hybrid Credit Bureau) Admin Portal — a React 18 SPA built with Vite, TypeScript, Tailwind CSS, and shadcn/ui.

**Canonical SPA backend (local dev):** `server/` — **Fastify** + in-memory state seeded from `src/data/*.json`. The Vite dev server proxies `/api` to **http://127.0.0.1:8091** by default. Override the proxy target when starting Vite: `VITE_API_PROXY_TARGET=http://127.0.0.1:8090 npm run dev` (Unix) or set the env var in your shell on Windows.

**Alternate backend:** `backend/` — **Spring Boot** on port **8090** with SQLite (dev) / PostgreSQL (prod). It is **not** the default proxy target; contracts differ (verbs, pagination, RBAC). See `docs/technical/Canonical-Backend.md` and `docs/technical/SPA-Service-Contract-Drift.md`.

When the API is running, use real login: **`admin@hcb.com` / `Admin@1234`** (also `super-admin@hcb.com`, `viewer@hcb.com` with the same password). If `VITE_USE_MOCK_FALLBACK=true` and the API is unreachable, the SPA falls back to mock auth and mock data per service layer.

**Data products:** With the Fastify dev API, **Create product** (`POST /api/v1/products` with `approval_pending`) persists the row and **enqueues** a `type: product` item on `GET /api/v1/approvals` (`metadata.productId`). **Packet catalogue for the form:** **`GET /api/v1/products/packet-catalog`** returns `productCatalogPacketOptions` from **`src/data/data-products.json`** (each entry has Schema Mapper **`sourceType`**, **`category`**, and **`derivedFields`** for the Configure → Derived tab). Spring **`backend/`** exposes the same path from **`classpath:catalog/product-packet-catalog.json`** — keep that file in sync with the SPA JSON when the catalogue changes. The create/edit product UI shows **one row per distinct source type within a category**, excludes **`custom`** and **Synthetic / Test**, sorts selected **`packetIds`** in catalogue order, and drops the old reorder card. Each row’s primary label is the **human-readable source type** only (e.g. Bank); packet catalogue **descriptions** and **per-packet subtitles** are not shown on the row. **One Configure control per row** opens **`PacketConfigModal`** for **all selected catalogue packets** in that source-type group (catalogue order). When several packets share the type, the modal shows a **Packet** switcher (catalogue labels); **Save configuration** persists **`packetConfigs`** for **each** packet in the group. **Raw** paths come from **`GET /api/v1/schema-mapper/schemas/source-type-fields?sourceType=`** merged with packet-only catalogue fields; **Sources** lists registry entries from **`GET /api/v1/schema-mapper/schemas?sourceType=`** (via **`useSourceTypeFields`** / **`useSchemaRegistryList`**). **Derived** field names come from each catalogue row’s **`derivedFields`** (API response or static seed); **`PacketConfigModal`** receives **`catalogOptions`** from the form so lookups match **`GET /api/v1/products/packet-catalog`**. The modal uses an **sr-only** dialog title; visible chrome is **Source type** badge, optional **Packet** switcher, **Sources**, and **Raw / Derived** tabs. **Consortiums:** **Create consortium** (`POST /api/v1/consortiums`, default or explicit `approval_pending`) enqueues `type: consortium` (`metadata.consortiumId`). The wizard **Members** step loads the add-member list with **`GET /api/v1/institutions?page=0&size=200&role=subscriber`** and **`allowMockFallback: false`** (subscribers and dual-role only; no mock catalogue). **Register member** (`POST /api/v1/institutions`) enqueues `type: institution` (`metadata.institutionId`); the member list requests **size=200** so new pending rows are included. After submit, the app returns to **`/institutions`** and invalidates institutions + approvals. Use `VITE_USE_MOCK_FALLBACK=false` and `npm run server` so create/queue actions hit the real API.

**Register member wizard:** Step 1 (**Entity, Regulatory, Contact, Participation, Consortium**) is **driven by backend geography configuration**, not hard-coded fields. **Step 3 (Review)** renders field values with the same **`text-body`** typography as the participation/consortium summary; value `className` uses a **template literal** (not **`cn(text-body, text-foreground)`**) so **`tailwind-merge`** does not drop **`text-body`** when merging with **`text-foreground`**. **`GET /api/v1/institutions/form-metadata?geography=<id>`** returns **`geographyId`**, **`geographyLabel`**, **`geographyDescription`**, **`registerForm.sections`** (each field: **`name`**, **`apiKey`** for the create payload, **`label`**, **`inputType`**, **`required`**, **`maxLength`**, **`placeholder`**, **`selectionMode`** / **`inputType`** for **single** vs **multi** select, **`options`** or resolved **`optionSource`** such as **`institutionTypes`** | **`activeConsortiums`**), plus **`institutionTypes`**, **`activeConsortiums`**, **`requiredComplianceDocuments`**. The canonical config file in dev is **`src/data/institution-register-form.json`** (per-geography **`sections`**); production would load the same shape from tenant/region config. **`POST /api/v1/institutions?geography=<id>`** validates the body against that geography’s register form (enums, lengths, required fields). Set **`VITE_INSTITUTION_REGISTER_GEOGRAPHY`** so the SPA’s metadata query and create call use the same geography. **`institutionTypes`** is seeded from **`src/data/institutions.json`**; **`activeConsortiums`** lists consortiums with **`status`** **`active`**. **`requiredComplianceDocuments`** comes from **`institutions.json`**; when **`null`** or empty after normalisation, the SPA **skips the compliance documents step**. Document rows support **`documentName`**, **`label`**, **`hint`**, **`maxSizeBytes`**, **`accept`**, **`requiredWhen`**: **`data_submitter`** | **`subscriber`**. **`POST /api/v1/institutions`** accepts optional **`consortiumIds`**; the SPA sends them only when **Subscriber** is checked. Mock fallback builds **`registerForm`** client-side when the API is unreachable and **`VITE_USE_MOCK_FALLBACK=true`**.

**Member traffic lifecycle:** Data Submission API, batch ingestion, and Enquiry API (and **`POST /api/v1/batch-jobs/:id/retry`** when the job has **`institution_id`**) require the member institution to be **`active`**. **Pending**, **suspended**, **draft**, or other non-active states return **403** with **`ERR_INSTITUTION_*`** codes — see **`docs/technical/Global-API-Error-Dictionary.md`** §2.3 and **`docs/technical/Data-Flow-Diagram.md`**. Batch **cancel** does not apply this gate.

**Schema Mapper Agent:** Fastify exposes **`/api/v1/schema-mapper/*`** (ingest, mappings CRUD, rules, drift stub, metrics). **`GET /api/v1/schema-mapper/schemas/source-type-fields?sourceType=`** returns the union of ingested **`parsedFields`** paths for that Schema Mapper source type (used by **Data Governance → Validation Rules** create-rule expression field dropdown and by **Data Products → packet Configure → Raw data**). Dev seed and ingest templates use **`src/data/schema-mapper.json`** reference arrays **`telecomParsedFields`**, **`utilityParsedFields`**, **`bankParsedFields`**, **`gstParsedFields`**, **`customParsedFields`** (and matching **`*FieldStatistics`**). Wizard Step 1 **Source Type** and **Data Category** lists come from **`GET /api/v1/schema-mapper/wizard-metadata`** (**`wizardSourceTypeOptions`** / **`wizardDataCategoryOptions`** in the same JSON); **`fetchWizardMetadata`** with mock fallback when the API is unreachable. **`GET /api/v1/schema-mapper/schemas`** supports **`sourceType`** (and pagination) for the registry list used in **PacketConfigModal → Sources** and the main Schema Mapper registry UI. Mapping generation runs **asynchronously** (~400ms dev worker) with heuristic output; set **`OPENAI_API_KEY`** (and optional **`OPENAI_SCHEMA_MODEL`**) for optional ChatGPT structured mapping. Disable LLM with **`SCHEMA_MAPPER_LLM_ENABLED=false`**. **`POST /api/v1/schema-mapper/mappings/:id/submit-approval`** enqueues **`type: schema_mapping`** with **`metadata.mappingId`**; approve/reject/request-changes updates in-memory mapping status. With **`VITE_USE_MOCK_FALLBACK=false`**, the Schema Mapper wizard and registry call these APIs.

**Data Ingestion Agent (drift alerts):** In-memory store **`state.ingestionDriftAlerts`** is seeded from **`src/data/data-governance.json`** `driftAlerts`. **`GET /api/v1/data-ingestion/drift-alerts`** (JWT) supports **`dateFrom`**, **`dateTo`**, **`sourceType`** (matches Schema Mapper registry `sourceType` via source name overlap). **`POST /api/v1/schema-mapper/ingest`** appends a **schema** alert; the async mapping worker appends a **mapping** alert when a job completes. The Data Quality Monitoring page uses **`data-ingestion.service.ts`** / **`useDriftAlerts`**; with **`VITE_USE_MOCK_FALLBACK=true`** and API errors, the client filters the same JSON mock as before. Schema Mapper ingest / mapping mutations invalidate drift-alert queries so the list refreshes.

### Key commands

See `package.json` scripts for the canonical list:

- **Dev server (frontend):** `npm run dev` — starts Vite on port 8080
- **API server (backend):** `npm run server` — `tsx watch server/src/index.ts` on port **8091** (set `PORT` to override)
- **Lint:** `npm run lint` — runs ESLint (expect pre-existing warnings/errors in the repo; these are not regressions)
- **Test:** `npm run test` — runs Vitest
- **Build:** `npm run build` — production build via Vite

### Non-obvious notes

- The lockfile is `package-lock.json`; use **npm** (not pnpm/yarn/bun) even though `bun.lock` / `bun.lockb` also exist in the repo.
- `npm run lint` exits non-zero due to pre-existing ESLint errors (e.g. `@typescript-eslint/no-explicit-any`, `@typescript-eslint/no-empty-object-type`). These are not regressions — do not try to fix them unless specifically asked.
- The Vite dev server binds to `::` (all interfaces) on port **8080** (configured in `vite.config.ts`).
- **Production:** set `JWT_SECRET` on the API host; never ship the default dev secret.
- No Docker or external DB required for local dev (API uses in-memory state).
