# AGENTS.md

## Cursor Cloud specific instructions

### Product overview

HCB (Hybrid Credit Bureau) Admin Portal — a React 18 SPA built with Vite, TypeScript, Tailwind CSS, and shadcn/ui.

**Canonical SPA backend (local dev):** `backend/` — **Spring Boot** on port **8090** (override with `SERVER_PORT`) with **SQLite** at `backend/data/hcb_platform.db` by default (PostgreSQL in prod). The Vite dev server proxies `/api` to **http://127.0.0.1:8090** by default. Copy `.env.development.example` → `.env.development` and set `VITE_USE_MOCK_FALLBACK=false` for durable API-backed dev.

**Legacy dev API:** `server/` — **Fastify** + in-memory state (port **8091** via `npm run server` / `server:legacy`). Use only for comparing old behaviour or running legacy Node-only tests; it is **not** the product contract owner. See `docs/technical/Canonical-Backend.md` and `docs/technical/SPA-Service-Contract-Drift.md`.

When the API is running, use real login: **`admin@hcb.com` / `Admin@1234`** (also `super-admin@hcb.com`, `viewer@hcb.com` with the same password). If `VITE_USE_MOCK_FALLBACK=true` and the API is unreachable, the SPA falls back to mock auth and mock data per service layer.

**Data products:** With the Spring API (and previously with Fastify), **Create product** (`POST /api/v1/products` with `approval_pending`) persists the row and **enqueues** a `type: product` item on `GET /api/v1/approvals` (`metadata.productId`). **Packet catalogue for the form:** **`GET /api/v1/products/packet-catalog`** returns `productCatalogPacketOptions` from **`src/data/data-products.json`** (each entry has Schema Mapper **`sourceType`**, **`category`**, and **`derivedFields`** for the Configure → Derived tab). Spring **`backend/`** exposes the same path from **`classpath:catalog/product-packet-catalog.json`** — keep that file in sync with the SPA JSON when the catalogue changes. The create/edit product UI shows **one row per distinct source type within a category**, excludes **`custom`** and **Synthetic / Test**, sorts selected **`packetIds`** in catalogue order, and drops the old reorder card. Each row’s primary label is the **human-readable source type** only (e.g. Bank); packet catalogue **descriptions** and **per-packet subtitles** are not shown on the row. **One Configure control per row** opens **`PacketConfigModal`** for **all selected catalogue packets** in that source-type group (catalogue order). When several packets share the type, the modal shows a **Packet** switcher (catalogue labels); **Save configuration** persists **`packetConfigs`** for **each** packet in the group. **Raw** paths come from **`GET /api/v1/schema-mapper/schemas/source-type-fields?sourceType=`** merged with packet-only catalogue fields; **Sources** lists registry entries from **`GET /api/v1/schema-mapper/schemas?sourceType=`** (via **`useSourceTypeFields`** / **`useSchemaRegistryList`**). **Derived** field names come from each catalogue row’s **`derivedFields`** (API response or static seed); **`PacketConfigModal`** receives **`catalogOptions`** from the form so lookups match **`GET /api/v1/products/packet-catalog`**. The modal uses an **sr-only** dialog title; visible chrome is **Source type** badge, optional **Packet** switcher, **Sources**, and **Raw / Derived** tabs. **Consortiums:** **Create consortium** (`POST /api/v1/consortiums`, default or explicit `approval_pending`) enqueues `type: consortium` (`metadata.consortiumId`). The wizard **Members** step loads the add-member list with **`GET /api/v1/institutions?page=0&size=200&role=subscriber`** and **`allowMockFallback: false`** (subscribers and dual-role only; no mock catalogue). **Register member** (`POST /api/v1/institutions`) enqueues `type: institution` (`metadata.institutionId`); the member list requests **size=200** so new pending rows are included. After submit, the app returns to **`/institutions`** and invalidates institutions + approvals. Use `VITE_USE_MOCK_FALLBACK=false` and `npm run spring:start` (or an already-running Spring process) so create/queue actions hit the real API.

**Register member wizard:** Step 1 (**Entity, Regulatory, Contact, Participation, Consortium**) is **driven by backend geography configuration**, not hard-coded fields. **Step 3 (Review)** renders field values with the same **`text-body`** typography as the participation/consortium summary; value `className` uses a **template literal** (not **`cn(text-body, text-foreground)`**) so **`tailwind-merge`** does not drop **`text-body`** when merging with **`text-foreground`**. **`GET /api/v1/institutions/form-metadata?geography=<id>`** returns **`geographyId`**, **`geographyLabel`**, **`geographyDescription`**, **`registerForm.sections`** (each field: **`name`**, **`apiKey`** for the create payload, **`label`**, **`inputType`**, **`required`**, **`maxLength`**, **`placeholder`**, **`selectionMode`** / **`inputType`** for **single** vs **multi** select, **`options`** or resolved **`optionSource`** such as **`institutionTypes`** | **`activeConsortiums`**), plus **`institutionTypes`**, **`activeConsortiums`**, **`requiredComplianceDocuments`**. The canonical config file for the **SPA** is **`src/data/institution-register-form.json`**; Spring serves the same shape from **`backend/src/main/resources/config/institution-register-form.json`** — keep them in sync when editing layouts. **`POST /api/v1/institutions?geography=<id>`** validation on Spring may still be narrower than Fastify; align incrementally. Set **`VITE_INSTITUTION_REGISTER_GEOGRAPHY`** so the SPA’s metadata query and create call use the same geography. **`institutionTypes`** is seeded from **`src/data/institutions.json`**; **`activeConsortiums`** lists consortiums with **`status`** **`active`**. **`requiredComplianceDocuments`** comes from **`institutions.json`**; when **`null`** or empty after normalisation, the SPA **skips the compliance documents step**. Document rows support **`documentName`**, **`label`**, **`hint`**, **`maxSizeBytes`**, **`accept`**, **`requiredWhen`**: **`data_submitter`** | **`subscriber`**. **`POST /api/v1/institutions`** accepts optional **`consortiumIds`**; the SPA sends them only when **Subscriber** is checked. Mock fallback builds **`registerForm`** client-side when the API is unreachable and **`VITE_USE_MOCK_FALLBACK=true`**.

**Member traffic lifecycle:** Data Submission API, batch ingestion, and Enquiry API (and **`POST /api/v1/batch-jobs/:id/retry`** when the job has **`institution_id`**) require the member institution to be **`active`**. **Pending**, **suspended**, **draft**, or other non-active states return **403** with **`ERR_INSTITUTION_*`** codes — see **`docs/technical/Global-API-Error-Dictionary.md`** §2.3 and **`docs/technical/Data-Flow-Diagram.md`**. Batch **cancel** does not apply this gate.

**Batch Execution Console (Spring):** **`GET /api/v1/batch-jobs/{id}/detail`** returns **`phases`**, **`stages`**, **`flowSegments`**, **`logs`**, and **`errorSamples`** (camelCase) when **`batch_phase_logs`** rows exist for that job; otherwise legacy flat **`stages`**. Tables: **`batch_phase_logs`**, **`batch_stage_logs`**, **`batch_error_samples`** in **`create_tables.sql`**. SPA: **`resolveBatchConsoleData`** (`src/lib/batch-console-from-api.ts`) + **`useBatchDetail`**; mock **`batchConsoleByBatchId`** still applies to **`BATCH-*`** ids from static monitoring JSON. Seeded demo job **`999901`** has a full multi-phase tree.

**Monitoring / reporting status pills:** Spring exposes **lowercase** enum strings for API requests, enquiries, and reports. **`src/lib/status-badges.ts`** normalizes keys, applies semantic **`Badge`** colours, human-readable labels, and **filter `Select` values** aligned with **`MonitoringController`**’s **`lower(column) = lower(?)`** checks (**`API_REQUEST_STATUS_FILTER_OPTIONS`**, **`ENQUIRY_STATUS_FILTER_OPTIONS`**). **`ReportListPage`** uses **`reportStatusesEqual`** so **`queued`** from the API matches filter **`Queued`**.

**Approval queue (Spring):** **`GET /api/v1/approvals`** returns paged items with string **`id`** and **`metadata`** (`institutionId`, `mappingId`, `productId`, `consortiumId`, `alertRuleId` — whichever applies) so the Approval Queue UI and Schema Mapper deep link work without mock-only shapes. **`POST /api/v1/approvals/{id}/approve`**, **`/reject`**, and **`/request-changes`** return **204 No Content** (empty body). Rows are inserted when: **register member** (`POST /v1/institutions`), **create product** with **`status`** **`approval_pending`**, **`pending`**, or **`pending_approval`** (stored as **`pending_approval`** in DB), **create consortium** unless the body sends **`status: active`**, **create alert rule** (rule row **`alert_rule_status=pending_approval`**; **`GET /v1/alert-rules`** maps to UI strings **`Pending approval`**, **`Enabled`**, **`Disabled`**), or **schema mapper submit-approval**. **`ApprovalQueueService`** performs the shared **`approval_queue`** insert. Approving/rejecting/requesting changes updates the underlying institution, product, consortium, alert rule, or schema-mapping state per type. **`POST /v1/alert-rules/:id/activate`** returns **400** while the rule is still **`pending_approval`**. If a dev SQLite file was created before **`approval_item_type` included `alert_rule`** or before **`pending_approval`** on alert rules, delete **`backend/data/hcb_platform.db`** (and **`-shm`/`-wal`**) and restart so **`spring.sql.init`** reapplies **`create_tables.sql`**. The SPA **`api-client`** treats **empty success bodies** as **`undefined`** so **204** and legacy **200**-empty responses do not break **`res.json()`**.

**Schema Mapper Agent:** **Spring** (`backend/`, default **8090**) owns **`/api/v1/schema-mapper/*`** — ingest, registry/schemas, **`wizard-metadata`**, async mappings (POST **202**), rules, drift stub, metrics, **`submit-approval`** (inserts **`approval_queue`** with **`approval_item_type=schema_mapping`**, **`entity_ref_id`** = mapping id; approve/reject/request-changes update persisted mapping + registry JSON). Persistence: **`schema_mapper_*`** tables + classpath **`config/schema-mapper.json`** (keep in sync with **`src/data/schema-mapper.json`**). **`GET /api/v1/schema-mapper/schemas/source-type-fields?sourceType=`** returns the union of **`parsedFields`** paths (used by **Validation Rules** and **Data Products → PacketConfigModal → Raw data**); **400** if `sourceType` missing or **`all`**. **Member institution** pickers use **`InstitutionFilterSelect`**: **`GET /api/v1/institutions`** with **`role=dataSubmitter`**, **`role=subscriber`**, or unscoped, **`allowMockFallback: false`**. Option labels use **`institutionDisplayLabel`** (`src/lib/institutions-display.ts`): **legal `name`** first, then **`tradingName`** if legal is empty. Wizard Step 1 uses that control for **source name**; **`fetchWizardMetadata`** for dropdowns (no inline GET path under **Source Type**). Dashboard command-center **`member`** strings from Spring use the same precedence (`DashboardController` SQL). **`GET /api/v1/schema-mapper/schemas`** supports **`sourceType`** and pagination (server caps **`size`** at **500**). Mapping worker ~**400ms** then heuristics + optional OpenAI: **`OPENAI_API_KEY`**, **`hcb.schema-mapper.openai-model`** / **`OPENAI_SCHEMA_MODEL`**, disable with **`hcb.schema-mapper.llm-enabled=false`**. **`hcb.schema-mapper.enabled`** defaults **true**; H2 tests without DDL set it **false**. **Legacy Fastify** (`server/`, **8091**) is in-memory reference only. **LLM Field Intelligence:** **`containsPii`** on **`fieldMappings`** via **`PATCH …/mappings/:id`** (`schema-mapper-api.ts`). With **`VITE_USE_MOCK_FALLBACK=false`**, the SPA calls Spring.

**Data Ingestion Agent (drift alerts):** **Spring** serves **`GET /api/v1/data-ingestion/drift-alerts`** (`DataIngestionController`) from table **`ingestion_drift_alerts`** (seeded in **`seed_data.sql`**, aligned with **`data-governance.json`**). Query params **`dateFrom`**, **`dateTo`**, **`sourceType`**; **`sourceType`** filtering uses **`schema_mapper_registry`** payloads when present (Fastify-equivalent semantics). **Legacy Fastify** keeps in-memory **`state.ingestionDriftAlerts`**. **`POST /api/v1/schema-mapper/ingest`** / mapping completion may append alerts on Java per **`SchemaMapperController`** implementation; invalidate drift queries after mapper mutations. The Data Quality Monitoring page uses **`data-ingestion.service.ts`** / **`useDriftAlerts`**; with **`VITE_USE_MOCK_FALLBACK=true`** and API errors, the client can still filter the JSON mock.

### Key commands

See `package.json` scripts for the canonical list:

- **Dev server (frontend):** `npm run dev` — starts Vite on port 8080 (proxies `/api` → Spring **8090** by default)
- **API server (Spring):** `npm run spring:start` — `mvn -f backend/pom.xml spring-boot:run` (requires **Maven** on `PATH`; SQLite DB under `backend/data/` unless `HCB_DB_PATH` is set)
- **API server (legacy Fastify):** `npm run server` or `npm run server:legacy` — port **8091** (`PORT` to override)
- **Backend tests:** `npm run spring:test` — `mvn -f backend/pom.xml test`
- **Route parity smoke (heuristic):** `npm run check:route-parity` — scans Fastify route strings vs Spring `*Controller` mappings (`scripts/route-parity-check.mjs`; noisy, not a strict contract)
- **Lint:** `npm run lint` — runs ESLint (expect pre-existing warnings/errors in the repo; these are not regressions)
- **Test:** `npm run test` — runs Vitest (client + small **server/** unit tests; no Fastify HTTP integration suite)
- **Build:** `npm run build` — production build via Vite

### Non-obvious notes

- The lockfile is `package-lock.json`; use **npm** (not pnpm/yarn/bun) even though `bun.lock` / `bun.lockb` also exist in the repo.
- `npm run lint` exits non-zero due to pre-existing ESLint errors (e.g. `@typescript-eslint/no-explicit-any`, `@typescript-eslint/no-empty-object-type`). These are not regressions — do not try to fix them unless specifically asked.
- The Vite dev server binds to `::` (all interfaces) on port **8080** (configured in `vite.config.ts`).
- **Production:** set `JWT_SECRET` on the API host; never ship the default dev secret.
- No Docker required for local dev; Spring dev uses **SQLite** on disk (`backend/data/hcb_platform.db`). Legacy Fastify uses in-memory state only.
- **SQLite seed time windows:** `backend/src/main/resources/db/seed_data.sql` anchors operational rows to a fixed calendar band (see the seed header). Endpoints that filter with `datetime('now', '-30 days')` (dashboard, monitoring KPIs, institution overview charts) show little or no data if your machine clock is far outside that band. For a clean re-init, stop Spring, delete `backend/data/hcb_platform.db` (and `-shm`/`-wal` if present), and start again so `spring.sql.init` reapplies schema + seed. Dev datasource **Hikari `maximum-pool-size` is 5** so JPA (open-in-view) and `JdbcTemplate` in the same request do not deadlock on a single connection.
- **List APIs vs DDL:** Spring **`JdbcTemplate`** for **consortiums**, **products**, **reports**, **sla-configs**, **alert-rules**, **users**, and **audit-logs** must stay aligned with **`backend/src/main/resources/db/create_tables.sql`** — mismatches surface as **`500` / `ERR_INTERNAL`**. **`GET /api/v1/audit-logs`** is denied for role **`VIEWER`** (`SecurityConfig`); use **ANALYST+** or **API_USER** for the Activity Log API. Details: **`docs/technical/SPA-Service-Contract-Drift.md`**, **`Canonical-Backend.md`**, **TDL-018** in **`Technical-Decision-Log.md`**.
